Cash on hand (start): derived for both entities as end + disbursements − receipts, never read from the filed field. Candidates carry no cash_on_hand_beginning_period at all; committees report one, but reading it can break the penny-tie on very high-volume filers (ActBlue's filed opening is ~$889 off the cash identity on $3.8B), so we derive in both cases and keep the filed Raised / Spent / ending-cash figures exact. Disclosed via an info tooltip beside the “Money flow” title: “Total raised, total spent, and ending cash are shown exactly as filed. Opening cash is derived from those figures. For nearly all committees this matches the filed opening balance; for very high-volume filers, whose monthly reports carry small rounding differences, it can vary by a fraction of a percent.”

‡ Candidate self-funding (candidate committees only) groups the candidate's own gift (candidate_contribution) with their own loan (loans_made_by_candidate) by source, so a self-funder isn't hidden inside Loans — which is then external only (candidate all_other_loans, committee all_loans_received). Because candidate_contribution is part of the contributions aggregate, the Contributions group node sums only the individual / PAC / party leaves, never the contributions total (which would double-count it).

Decisions log

Settled

  • Conserving model — cash-on-hand (start) + receipts = disbursements + cash-on-hand (end). Both sides tie to the reported Raised / Spent to the penny.
  • Debt — an annotation, never a flow node. It's a balance-sheet stock; loan activity already appears as Loans-in and Loan-repayments.
  • Cash-on-hand — neutral grey, placed at the bottom of each column; drawn only when non-zero (a $0 start or end isn't rendered).
  • Cash-on-hand (start) — derived — computed for both entities as end + disbursements − receipts, never read from the filed field. Candidates have no opening field; committees do, but the filed value can miss the cash identity on very high-volume filers (ActBlue: ~$889 on $3.8B), so deriving keeps the chart conserving while Raised / Spent / ending cash stay exactly as filed. An info tooltip beside the “Money flow” title discloses it. Distinct from the dual-account gate below — this is a cash-identity fix, that is a leaf-coverage gate.
  • Contra-flows (offsets in, refunds out) — rendered as ordinary nodes, ranked by size like any category; no visual distinction. Not netted out — conservation holds because they're already gross leaves in the FEC partition (the net_* fields are excluded).
  • Color — assigned by size rank within each column (largest = darkest, descending), using the design system's --cat-1…--cat-7 sequence. Applied ordinally by rank, not by category identity; node order locked to size.
  • Other categories — Other receipts / Other disbursements use --cat-other (the system's neutral) and are pinned to the bottom of the category stack, directly above cash-on-hand, regardless of size; excluded from the rank ramp.
  • Node labels — name + dollar value on desktop; name only on mobile (below 640px).
  • Node stroke — transparent. The earlier 1px parchment-colored border left each colored block slightly inset from its flow ribbon (a visible seam); transparent makes node and path meet flush.
  • Minimum node height — ECharts has no native minimum, so a genuinely tiny node (e.g. $12 of $2.4M) rendered at sub-pixel height and vanished while its label/flow stayed. Each node's layout height is floored to 0.5% of its column so it stays visible and hoverable; the true value drives the label and tooltip. Negligible categories are thus slightly over-represented in height — the underlying data still conserves exactly.
  • Contributions — the subgroups (individuals, PACs, party) always share one color and sit together as a contiguous block; the block is ranked among the other categories by its combined total.
  • Candidate self-funding — on candidate committees, the candidate's own gift (candidate_contribution) and own loan (loans_made_by_candidate) form one source, so a self-funder isn't hidden inside Loans (which becomes external-only). Candidate committees only. The Contributions group sums the individual / PAC / party leaves, not the contributions aggregate, so the gift isn't double-counted.
  • Committee — a single slate hub (commingled funds).
  • Responsive — one full view at all widths; no mobile flatten.
  • Hover — adjacency focus; non-focused elements recede gently rather than the library's near-disappear default. Blur opacities: flows 0.40 (from a 0.48 base), nodes 0.50, labels 0.85.
  • Tooltip — node-hover and flow-hover share one format: the category label, then amount • percent (share of the side's receipts or disbursements). Cash-on-hand and the committee hub show amount only — no clean percent denominator.
  • Spacing — node gap 16px.
  • Library — Apache ECharts sankey series (self-hostable single file).

v1 scope gates — not modeled in v1

  • Dual-account committees — committees that run a non-federal (state/local soft-money) account fold non-federal money into the receipts / disbursements totals the chart conserves against (Dem Party of Wisconsin Fed, C00019331: +$12.1M in / +$15.3M out vs the federal figures — ~19% / ~25%). The v1 federal leaf set would under-sum, so these render a neutral “not yet modeled” state in the Money-flow slot — never an under-summing chart, never a donut fallback. Detector fires when |receipts − fed_receipts| or |disbursements − fed_disbursements| exceeds $1 (committee entities only — candidate totals have no fed_* fields). Not committee type: DCCC (national party) is fed-only and renders normally. Modeling the non-federal dimension is a fast-follow.
  • Presidential committees (Form 3P) — carry extra leaves the v1 model doesn't render (federal_funds, fundraising_disbursements, exempt_legal_accounting_disbursement), so they're gated to the same “not yet modeled” state. Fast-follow.

Still open

  • Color application model — size rank (current) vs. the design system's category-identity mapping (a category always = its fixed --cat, matching the donut). Identity would align Sankey↔donut color but breaks size-rank ordering and the contributions-as-one-color grouping.
  • Light-end legibility — --cat-6/--cat-7 are pale, so the smallest-rank flows read faint at 0.48 opacity (the node rectangle still carries the solid color).
  • Label legibility where dark text sits over a dark flow band.
  • Accessibility — a visually-hidden data table mirroring the flows.
  • Spending-purpose layer (Schedule B) — deferred to Phase 4.
