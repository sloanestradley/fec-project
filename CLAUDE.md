## What this is

A web-based campaign finance visualization tool built on the FEC public API. The goal: give political strategists, journalists, and researchers a faster, clearer window into where money is flowing in a race than the FEC website provides.

This is also a portfolio piece for a staff-level product designer (Sloane). It needs to look and feel like a designer built it ŌĆö not a developer prototype.

## Tech stack

- Vanilla HTML/CSS/JS ŌĆö no framework, intentional for this stage
- Chart.js 4.4.0 + chartjs-adapter-date-fns 3.0.0 (time scale support)
- Google Fonts: Oswald (display/headings) + IBM Plex Sans (body/nav) + IBM Plex Serif (editorial/narrative prose) + IBM Plex Mono (labels/data)
- FEC public API: `https://api.open.fec.gov/v1`
- Cloudflare Pages Functions for server-side logic ŌĆö proxy at `functions/api/fec/[[path]].js`, routing at `functions/candidate/[[catchall]].js` and `functions/committee/[[catchall]].js`
- No build step ŌĆö files are served directly
- **Clean URLs:** `_redirects` defines Netlify 200 rewrites for all pages. Profile pages with path-segment URLs (`/candidate/:id`, `/committee/:id`) **must use absolute paths** for every local resource and nav link ŌĆö `href="/styles.css"`, `src="/main.js"`, `href="/candidates"`, etc. Relative paths break because the browser treats the path segment as a subdirectory (e.g. from `/candidate/H2WA03217`, relative `utils.js` resolves to `/candidate/utils.js`, which also matches the rewrite rule and returns HTML served as JS). Browse pages (`/candidates`, `/committees`, `/races`, `/race`, `/search`) use single-level paths so relative links still resolve to root ŌĆö but any new page with a deeper path must follow the absolute-path rule.
- **Profile-page ID extraction (canonical pattern):** Use `pathname.split('/').filter(Boolean)` then take the last segment, AND exclude both `.html` and clean-URL page name from the ID guard: `urlParams.get('id') || (_pathId && _pathId !== 'committee.html' && _pathId !== 'committee' ? _pathId : null)`. Two regression instances at this point: trailing slash (`/committee/{id}/` ŌåÆ `.pop()` returns `''` without `filter(Boolean)`) and page-name-as-ID (`/committee` ŌåÆ `_pathId='committee'` extracted as a literal ID, fires a 404 fetch instead of the friendly "no ID provided" branch). Both pages ŌĆö candidate.html and committee.html ŌĆö must include both guards. Friendly no-ID error state links to the corresponding browse page: `'No {entity} ID provided. <a href="/{browse}">Browse {entities} ŌåÆ</a>'`.
- **Testing:** Playwright (`@playwright/test`) ŌĆö `npx playwright test` runs 904 structural tests (mocked API); `npm run test:smoke` runs 5 live-API smoke tests against `https://fecledgerapp.pages.dev`. See `TESTING.md`.
- **apiFetch concurrency queue:** `utils.js` implements a `MAX_CONCURRENT = 4` request queue to avoid 429 rate-limit errors when pages fire many parallel API calls (candidate page fires 15ŌĆō20 on load). All calls still execute ŌĆö they just pace to Ōēż4 in-flight at a time. No call-site changes needed; `apiFetch(path, params)` signature is identical.
- **FEC API key is server-side (Cloudflare secret):** `utils.js` sends requests to `/api/fec/*`, which proxies to `api.open.fec.gov/v1/*` via a Cloudflare Pages Function at `functions/api/fec/[[path]].js`. The API key is stored as a Cloudflare secret (`API_KEY`) and injected server-side ŌĆö it is no longer visible in client-side source. All visitors draw from the same rate limit. An upgraded key is confirmed at 7,200 calls/hour (120 calls/min). **Live site:** `https://fecledgerapp.pages.dev` (Cloudflare Pages ŌĆö see the Deployment note above).
- **FEC API field verification:** Before writing logic that depends on a specific field name or value from any FEC endpoint, verify the actual response shape first. Navigate directly to the endpoint in a browser (or use `apiFetch` in the console) and confirm field names, value formats, and null behavior. Do not infer from the FEC docs alone ŌĆö the docs and actual responses diverge in practice (e.g. `/elections/` returns `incumbent_challenge_full` as `"Incumbent"/"Challenger"/"Open seat"`, not the single-letter `incumbent_challenge` code). Document any verified field behavior in the relevant section below.
- **Cloudflare KV for pre-computed aggregations:** Namespace `fecledger-aggregations` stores top-25 contributors per committee per cycle, populated daily by `scripts/precompute-aggregations.js` (see pre-computation architecture below). Bound to the Pages project as `AGGREGATIONS` ŌĆö Pages Functions read via `env.AGGREGATIONS.get('top_contributors:{cmte_id}:{cycle}')`. The binding is **configured manually in the Cloudflare dashboard** (Workers & Pages ŌåÆ fecledgerapp ŌåÆ Settings ŌåÆ Functions ŌåÆ KV namespace bindings); it cannot be set via wrangler for Pages projects. Writes happen from GitHub Actions via the Cloudflare REST API (bulk PUT/DELETE/list endpoints); reads happen from Pages Functions via the binding.
- **DuckDB in the aggregation pipeline:** `scripts/precompute-aggregations.js` uses `@duckdb/node-api` (Neo binding) to run a single SQL GROUP BY over the indiv CSV. DuckDB spills to disk automatically when memory is exhausted, giving bounded memory + 100% accurate totals. An earlier streaming-Map approach with mid-stream pruning was abandoned because pruning silently undercounts late-cycle contributors whose prior accumulation was discarded.
- **Verify input data before shipping pipeline SQL or schema changes:** Before writing code that depends on a specific column position, field name, value format, or quoting convention in any FEC bulk file, verify the actual file: fetch the FEC description page AND `head -n 2` (or `awk -F'|' '{print NF}'`) a real sample. Write the schema constant against what's in the file, not against what the docs or prior code assumed. This has been paid for three times: pas2 21-vs-22 columns (2026-04-17) broke DuckDB downstream with a cryptic parse error; pas2 NAME semantics (2026-04-20, recipient_name vs. giver_name) shipped to production with visually wrong data and required a same-day rollback; cm.txt (2026-04-21) was caught up front ŌĆö the real cm.txt file has literal `"` chars embedded in `CMTE_NM` that would have broken DuckDB's default `quote='"'`, the check against the real file surfaced the need for `quote=''` in 30 seconds of work. Cost of verifying up front: ~5 minutes. Cost of debugging after: a full pipeline re-run plus a follow-up commit. The discipline is also documented per-surface in the relevant strategy docs (see `strategy/cm-txt-integration.md` for the reference treatment).

## Design system

**Reference file:** `design-system.html` is the living design system reference. Read it (or at minimum the token table and component list) before building any new page or component.

**Skeleton loading:** `.skeleton` (in `styles.css`) is the standard placeholder for content that loads asynchronously after the initial page render. Use it whenever a UI element shows a loading state before data resolves ŌĆö set `width` and `height` inline to approximate the expected content size. Do not define page-specific skeleton keyframes; always use the shared class. Size guidance: height should match the resolved element's total height (content + padding), width should approximate the minimum resolved state. Do NOT wrap the skeleton in any resolved container that carries its own background, border, or padding during loading ŌĆö that adds a second visible layer behind the pulse. Wrapping in a layout-only container (no visual surface, e.g. `.race-context-line` post-2026-05-13 rename) is fine when the resolved container's flex/grid structure is needed to position the skeleton correctly.

**Tab-skeleton variants (T12):** `.tab-skeleton` is the block-level variant of `.skeleton` for chart/table cells inside Raised + Spent tabs on candidate.html and committee.html. Four sizes (defined in `styles.css`): `.skeleton-donut` (280px), `.skeleton-map` (360px), `.skeleton-table` (240px), `.skeleton-bars` (240px). Used with the **title-always-visible pattern**: the section title (`.raised-cell-title`, `.donors-head`) renders immediately and stays visible during the loading window; only the chart canvas / table body area swaps between skeleton and content via `display:none` / `display:block` toggling on per-section `*-skeleton` and `*-content` IDs. **`.inline-status-msg`** is the 10s "still working" augmentation under slow-tier skeletons (body type ŌĆö IBM Plex Sans 0.75rem muted, centered) ŌĆö fired by a `setTimeout` in `kickoffRaisedSlowFetch` / `kickoffSpentFetch` when the fetch is still pending at 10s. **On the Raised ŌåÆ Top Contributors sub-tabs (candidate + committee) the message is centered IN the skeleton's footprint via the `.skeleton-overlay-wrap` / `.skeleton-overlay-msg` overlay (T-raised-loading-states ŌĆö see the per-panel indicators note below), NOT below it; the Spent tab + map/empty states keep the under/detached placement.** Cleared on resolve, error, or cycle change. **Tab-error UI (T12 + T12.5):** `.tab-error` (in `styles.css`) is the per-tier error UI on Raised + Spent tabs. Three variants, selected by `showTabError(errorEl, err, defaultMsg)` ŌĆö **lifted to utils.js (shared) in T-race-inplace-cycle (2026-06-03)** from byte-identical inline copies on candidate.html + committee.html; race.html's inline cycle-switch error (`#race-switch-error`) is now a third consumer. The helper only sets copy + toggles the retry button's visibility; each page wires the retry button's own click handler (candidate/committee via `data-retry`, race via a closure over the failed cycle):
1. **Default** (network blip, server error, abort) ŌåÆ generic copy + retry button. Clicking retry re-fires the failed fetch via the `data-retry="raised-fast" | "raised-slow" | "spent"` attribute on the button (delegated event listener in each page's script block).
2. **429 rate-limit** ŌåÆ `is429(err)` matches `'FEC 429'` in the thrown error message (apiFetch in utils.js throws `'FEC ' + status + ' ŌĆö ' + path` on non-2xx). 3. **Init-stage failure** (candidate.html only) ŌåÆ `loadCycle()`'s outer catch sets `err.initStage = true` then sets `raisedFastError = raisedSlowError = spentOpexError = err` and calls `renderRaisedIfReady()` + `renderSpentDonutNow()` + `renderSpentOpexIfReady()` to surface the tab-error UI (post spent-progressive-loading the Spent error rides the opex tier; the donut skeleton is hidden via the null-breakdown guard in `renderSpentDonutNow`). Without this bridge, the kickoff functions never fire (they sit after the await that threw) and skeletons hold forever. Copy: `TAB_ERROR_INIT_FAILURE_MSG` ("ŌÜĀ Couldn't load this page. Please reload to try again."). Retry button hidden ŌĆö `currentPrincipalId` and other kickoff context wasn't populated, retry can't help; reloading is the recovery action.

Helper signatures:
- `is429(err)` (utils.js) ŌĆö boolean detector for the FEC 429 case.
- `TAB_ERROR_RATE_LIMIT_MSG` / `TAB_ERROR_INIT_FAILURE_MSG` (utils.js) ŌĆö single source of truth so copy edits don't drift.
- `showTabError(errorEl, err, defaultMsg)` (utils.js, shared across candidate.html + committee.html + race.html) ŌĆö picks the variant + sets DOM (copy + retry-button visibility). Retry click wiring stays per-page.

`apiFetch` itself does NOT auto-retry on 429. That's banked future polish (would honor `Retry-After` header from the proxy, retry up to 2 times with backoff, then surface to UI). For now, the user is expected to wait + reload per the rate-limit copy.

**CSS consolidation principle:** Component CSS lives in `styles.css`. Inline `<style>` blocks in individual pages are for page-specific overrides only (layout grid, page-specific spacing, page-specific components). `design-system.html` imports the same `styles.css` as production ŌĆö no component CSS is duplicated between pages.

**CSS scoping default:** When a global rule needs different behavior in a specific context, first consider whether the global rule itself should be updated (lifting) rather than adding a scoped override below it. Lifting takes several shapes: updating the global rule's value when the override is the "real" production value, moving the property to a more accurate selector (e.g. `margin-bottom` from `.stats-grid` to the strip wrappers that distinguish the two contexts), or deleting the utility along with its consumers when the rule's purpose is gone. Overrides accumulate and obscure intent. Lift before override is the default; scoped overrides are the exception when the global rule genuinely serves a different need elsewhere. Before proposing a scoped override, name out loud why the global rule shouldn't change instead.

**Design system demo override pattern:** Any shared component rule that uses `opacity:0` / `transition` for a JS-revealed animation needs a matching override in `design-system.html`'s inline `<style>` block, alongside the existing `.page-header-reveal` reset: `.ds-component-demo .component-class { opacity:1; transition:none; }`. Demos don't run page JS so `.visible` is never added ŌĆö without the override, the component renders invisible in the design system. **Banner and nav architecture (in-flow pattern):** `.global-banner` is in-flow (not fixed). `.top-nav` is `position:relative` ŌĆö also in-flow. Do not change this back to `hidden`. **The mobile-nav drawer is a DOM child of `<nav class="top-nav">`, not a sibling:** `#mobile-nav` (`<nav class="mobile-nav">`) lives inside `<nav class="top-nav">` and uses `position:absolute; top:100%`. `top:100%` on an absolutely-positioned child of the nav always resolves to directly below the nav's bottom edge. **Critical ŌĆö `/elections/` office param:** This endpoint requires `office` as a **lowercase full word** (`house`, `senate`, `president`), NOT the single-letter code (`H`, `S`, `P`) used by other endpoints. Passing `H`/`S`/`P` returns a 422 error. Use a conversion function:
```javascript
function officeApiParam(o) {
  return { H:'house', S:'senate', P:'president' }[o] || o.toLowerCase();
}
```
Other endpoints (`/candidates/`, `/candidate/{id}/totals/`) use the single-letter codes ŌĆö the inconsistency is an FEC API quirk.

**Critical ŌĆö `/elections/` party field:** This endpoint does NOT return a `party` field. Party affiliation comes back as `party_full` with full names like `"DEMOCRATIC PARTY"` / `"REPUBLICAN PARTY"`. The `partyClass(p, party_full)` and `partyLabel(p, party_full)` utilities in `utils.js` accept both fields and handle the cross-endpoint quirk internally via a dual-field design (see Shared files entry) ŌĆö pass `c.party` and `c.party_full` directly at every call site and the helpers produce a consistent label and class regardless of which endpoint fed the data. **Critical ŌĆö `/elections/` incumbent field:** This endpoint returns `incumbent_challenge_full` (e.g. `'Incumbent'`, `'Challenger'`, `'Open seat'`) ŌĆö NOT the short-code `incumbent_challenge: 'I'/'C'/'O'` that appears on the `/candidate/{id}/` metadata endpoint. The field is populated at time of candidacy filing, so it's available for future cycles as soon as a candidate has declared. Check `c.incumbent_challenge === 'I' || c.incumbent_challenge_full === 'Incumbent'` to handle both shapes (mock uses short code; live API returns full string).

**Critical ŌĆö `/elections/` returns `0.0`, NOT `null`, for non-filing candidates (verified live 2026-06-03):** A candidate who declared but never filed a financial report comes back with `total_receipts: 0.0` / `total_disbursements: 0.0` / `cash_on_hand_end_period: 0.0` ŌĆö **not** null and **not** absent. Verified live: WA-03 2024 has 1 such candidate, TX-18 2024 has 6, CA-12 2024 has 5. **Consequence:** any "did this candidate file?" / "has financial activity?" branch must use a **truthy** check (`if (c.total_receipts || c.total_disbursements)`), NOT `!= null` ŌĆö a `!= null` gate treats `0.0` as "has data" and never fires the no-data branch on real data. This bit T-race-page-UI's no-filings card state (built + tested against `null`, which the mock used, but silently never triggered live until the gate was changed to truthy). The trap generalizes to anything keying on filing status from `/elections/` (e.g. a future "unserious candidate" reduced-hierarchy treatment, race-level totals) ŌĆö non-filers are zeros, not nulls. ### Key FEC API field names (verified from live response)
Reports endpoint (`/committee/{id}/reports/`) returns per-filing objects with:
- `total_receipts_period` ŌĆö raised this filing period only
- `total_disbursements_period` ŌĆö spent this filing period only
- `total_receipts_ytd` ŌĆö cumulative raised, resets Jan 1 each year
- `total_disbursements_ytd` ŌĆö cumulative spent, resets Jan 1 each year
- `cash_on_hand_end_period` ŌĆö COH snapshot at end of period
- `coverage_start_date` / `coverage_end_date` ŌĆö in format `"2025-03-31T00:00:00"` (strip `T` and after)
- `report_form` ŌĆö e.g. `"Form 3"` (use this to filter deadlines)
- **Amended filings:** When multiple reports exist for the same period, use only the most recent. Amendment-tracking fields (verified from live response, C00806174 "Marie for Congress"):
  - `most_recent` (boolean) ŌĆö `true` = current authoritative version; `false` = superseded. **This is the correct dedup filter.** The API also accepts `?most_recent=true` as a query param to filter server-side.
  - `is_amended` (boolean) ŌĆö `true` = this record has been superseded by a newer filing. Equivalent to `most_recent: false`.
  - `amendment_indicator` ŌĆö `"N"` = originally filed as a new report; `"A"` = this record is itself an amendment filing
  - `amendment_indicator_full` ŌĆö `"NEW"` or `"AMENDMENT"`
  - `amendment_chain` ŌĆö array of `file_number` integers tracking the full amendment lineage
  - `most_recent_file_number` ŌĆö float; the `file_number` of the current authoritative version
  - **`amendment_version` does NOT exist** ŌĆö remove any logic relying on this field name; it is not present in API responses.

Candidate totals endpoint returns:
- `receipts` ŌĆö cycle total raised (sum of ALL receipt categories below)
- `disbursements` ŌĆö cycle total spent
- `last_cash_on_hand_end_period` ŌĆö most recent COH
- `coverage_end_date` ŌĆö most recent coverage date
- **Debt (verified live 2026-06-05):** exposed only as `last_debts_owed_by_committee` (+ `last_debts_owed_to_committee`), the end-of-cycle snapshot ŌĆö a point-in-time value, NOT summed across sub-cycles, correct for House / active Senate (any sub-cycle) / closed Senate / Presidential alike. **There is NO bare `debts_owed_by_committee` and NO `loans_received`** on this record (election_full or per-cycle) ŌĆö reading either returns undefined. Used by the closed-cycle banner debt tail; NOT a health-signal input (see banner note above).
- **Receipt breakdown fields** (all summed into `receipts`; used by "Raised breakdown" donut on candidate.html and committee.html ŌĆö both pages now surface the full 13-field breakdown):
  - `individual_itemized_contributions` ŌĆö itemized individual donations (>$200 required; Ōēż$200 permitted)
  - `individual_unitemized_contributions` ŌĆö lump-sum unitemized individual donations (always Ōēż$200)
  - `other_political_committee_contributions` ŌĆö PAC and other non-party committee contributions
  - `political_party_committee_contributions` ŌĆö contributions from official party committees
  - `transfers_from_other_authorized_committee` ŌĆö transfers between candidate-authorized committees

  - `loans_made_by_candidate` ŌĆö loans from candidate to own committee (creates repayable debt)
  - `all_other_loans` ŌĆö third-party loans (banks, etc.)
  - `federal_funds` ŌĆö presidential public financing only; always 0 for House/Senate
  - `offsets_to_operating_expenditures` ŌĆö vendor refunds credited back as receipts
  - `offsets_to_fundraising_expenditures` ŌĆö fundraising expense credits
  - `offsets_to_legal_accounting` ŌĆö legal/accounting expense credits

The three `offsets_to_*` fields are grouped as "Offsets" (label aligned 2026-06-10 across donuts + Sankey; was "Refunds & offsets"). The transfers-in fields render as "Transfers in" (was "Candidate authorized committees" on candidate). These fields are confirmed present (may be 0) on live responses.

Committee totals endpoint (`/committee/{id}/totals/`) ŌĆö amendment safety (verified from live response, C00806174 "Marie for Congress"):
- Returns one record **per cycle** ŌĆö 4 records for a multi-cycle committee, not one record by design. `per_page=1` with no cycle filter returns the most recent cycle only.
- Has **no amendment fields** (`is_amended`, `most_recent`, `amendment_indicator` are absent). The endpoint returns pre-aggregated cycle totals, not raw filings ŌĆö no dedup logic needed here.

Schedule A / Schedule B cursor pagination (verified live 2026-04-14):
- Both endpoints use cursor-based pagination, not page-numbered. The cursor is carried forward via `last_index` plus one sort-field-specific key returned in `response.pagination.last_indexes`.
- **Schedule A, sorted by `-contribution_receipt_amount`:** cursor key is `last_contribution_receipt_amount`. Used by Top Committee Contributors / Top Conduit Sources aggregation on both candidate.html and committee.html.
- **Schedule B, sorted by `-disbursement_amount`:** cursor key is **`last_disbursement_amount`** (NOT `last_disbursement_date` ŌĆö that was a pre-existing bug that caused 422 errors on any pagination beyond page 1, fixed 2026-04-14 across `fetchSpentData` (since split into `fetchSpentOpex` / `fetchSpentContributions`, 2026-06-05) in candidate.html and two Schedule B loops in committee.html).
- **Critical pattern:** the cursor field mirrors the sort field. If you change the sort key on either schedule, you must also update the cursor key. The mock at `tests/helpers/api-mock.js` returns `pagination: { count: N }` with no `pages` field, so cursor advance logic is bypassed in tests ŌĆö live verification before shipping is mandatory.
- **Broken endpoint:** `/schedules/schedule_a/by_contributor/` is documented in the FEC API spec but returns 500 Internal Server Error on reasonable queries (verified 2026-04-14 with `committee_id`, `cycle`, `two_year_transaction_period` combinations). Cannot be used for server-side contributor aggregation; the walk-and-aggregate pattern via `/schedules/schedule_a/` is the only working path today.

Elections-search endpoint (`/elections/search/`) returns:
- `cycle` ŌĆö integer, election cycle year (even number)
- `district` ŌĆö string, e.g. `'03'` (House only)
- `office` ŌĆö string, e.g. `'H'`, `'S'`, `'P'`
- `state` ŌĆö string, e.g. `'WA'`
- **Critical:** Returns projected future cycles out to 2060+ ŌĆö must cap client-side. House: cap at current cycle. Senate: cap at current cycle + 4 (covers both seats' next election).
- **Critical:** For Senate, returns cycles for *both* seats in the state (unioned). Deduplication required.
- **Critical:** No Senate class field exists anywhere in the FEC API (`/elections/`, `/elections/search/`). Senate seat class (I/II/III) must be derived heuristically from cycle year.

## Remaining architectural debt

- **Auto-retry-with-backoff in `apiFetch` (banked, T12.5 follow-up):** The Pages Function proxy at `functions/api/fec/[[path]].js` already passes through `Retry-After`, `x-ratelimit-limit`, `x-ratelimit-remaining` headers. `apiFetch` could read `Retry-After` on 429, schedule a retry up to 2 times with backoff (cap at ~5s for UX), then surface to UI. ## Unified browse+search architecture (candidates.html / committees.html)

Both browse pages use a single unified state machine ŌĆö no separate browse/search modes. Key patterns:

- **Auto-load on page visit** ŌĆö `doFetch(false)` fires in `init()` regardless of URL params. No "click to browse" gate.
- **Unified `doFetch(isLoadMore)`** ŌĆö single code path. Uses `activeQ` (string) and `activeFilters` (object) to build params. If `activeQ` is set, fires `Candidates/Committees Searched`; otherwise fires `Candidates/Committees Browsed`.
- **State vars:** `activeQ` (search query), `activeFilters` (state/office/party/cycle for candidates; state/type for committees), `currentPage`, `totalPages`, `loading`, `lastFetch` (fn ref for retry).
- **URL sync** ŌĆö `updateURL()` calls `pushState` after every fetch. `init()` restores from URL params on load.
- **Filter chips** ŌĆö `renderChips()` rebuilds chip row after every fetch. `clearFilter(key)` and `clearAllFilters()` reset state and re-fetch.
- **State combo** ŌĆö text input filters a `size="6"` listbox; `:focus-within` shows/hides the listbox. On selection, `f-state` fires `change`, populates `f-state-filter`, and calls `doFetch`.
- **Search field submit** ŌĆö Enter key or `#search-btn` click sets `activeQ` and calls `doFetch(false)`. There is no live typeahead ŌĆö the filter-field typeahead was retired in T-search-typeahead-retire (2026-05-21); both pages are enter-to-search. A query shorter than `FEC_MIN_KEYWORD_LENGTH` (3) is normalized to `''` at the top of `doFetch`, so a sub-threshold submit falls back to browse mode rather than 422-ing on the FEC API's keyword minimum.
- **All result links are clean URLs** ŌĆö `/candidate/{id}` and `/committee/{id}` in all modes (browse and search).
- **Error state** ŌĆö `#state-error` shown on API failure; `.retry-btn` calls `lastFetch()`.
- **`needsApiMock: true`** in `shared.spec.js` for both pages ŌĆö they make API calls on load.

- **FEC race layer (`races-resolver.js`) ŌĆö the /elections/ calls + seat-status.** `planRaces(geo, cycle)` builds the call list: House per district, Senate per state (DC skipped ŌĆö no voting Senate), President only in presidential cycles (`cycle % 4 === 0`, state=US); territory/error geo ŌåÆ `[]`. `fetchRaceSummary(item, cycle)` fires ONE `/elections/?per_page=200&sort=-total_receipts` call ŌåÆ a summary tile, or **`null` when a Senate/President call returns empty** ("office not up"); **House always returns a card** (biennial ŌĆö "No candidates reported" if empty, never null). `per_page=200` is exact-to-the-cent even for President (~115 funded of 869 filers in 2024). **Seat-status contract:** 0 candidates ŌåÆ "No candidates reported"; Ōēź1, 0 incumbents ŌåÆ "Open seat"; exactly 1 distinct incumbent ŌåÆ "Incumbent: {name}" (`formatCandidateName`, dedup by normalized name so a duplicated record isn't "Multiple"); 2+ distinct incumbents ŌåÆ "Multiple incumbents". - **Render (races.html inline).** Skeleton tiles paint immediately (race name shows; meta skeletons until each office's call lands), then **progressive per-card swap** (`fetchRaceSummary` per plan item: non-null ŌåÆ swap; null ŌåÆ drop, or Caption A note; error ŌåÆ drop). **Single-state ŌåÆ flat list; multi-state ZIP ŌåÆ per-state groups** (`buildGroupedSkeleton`) with **President ungrouped at the top** (national, belongs to no state). **Ordering (2c decisions, deliberately NOT geocod's proportion order):** House districts **numerical** by number (`orderPlan` ŌĆö for a ZIP crossing districts you live in exactly one; proportion order reads as arbitrary); multi-state groups **alphabetical by state name** (not majority-share). Multi-state ONLY ŌĆö single-state absent offices stay **silently dropped** (locked decision). **Wording describes the filing signal, never asserts seat class** (empty `/elections/` ŌēĀ "seat isn't up" ŌĆö the same unsafe inference we rejected for the collapse detector). **B:** the regular+special collapse note (`senateCollapseNote`, e.g. GA-2020) ships **dark behind `SENATE_COLLAPSE_ENABLED=false`** ŌĆö detecting a genuine collapse needs the term-class dataset (the `SG`+`G` shortcut false-positives on special-only cycles); one-flip to enable.

## Filing feed architecture (feed.html)

## Senate multi-sub-cycle architecture

Senate 6-year cycles introduce a multi-sub-cycle pattern worth understanding before modifying:

- `getSubCycles(cycle)` returns `[cycle-4, cycle-2, cycle]` ŌĆö three FEC 2-year periods
- **Cycle-level totals (`#stat-raised`/`#stat-spent`/`#stat-coh`, donut breakdowns):** single `/candidate/{id}/totals/?cycle={cycle}&election_full=true` call returns FEC's pre-aggregated record. **Was N parallel `?cycle={sc}` calls iterated and summed client-side until T-loadcycle-single-fetch (2026-05-19)** ŌĆö data parity verified across H/S/P samples (Gillibrand 2024 6yr, Trump 2024 4yr, Marie 2024 2yr) before the swap. The election_full record's COH and coverage_end_date represent end-of-cycle values, semantically aligned with the prior "most-recent subcycle" treatment.
- **Reports (chart per-period data):** still fetched per-subcycle via `getSubCycles` ŌĆö needs per-period granularity for the time-series chart, not election aggregation.
- **Schedule A/B (Raised + Spent tab data):** `fetchRaisedFastData` (one call, client-filtered to `subCycles`) + `fetchRaisedSlowData` (iterated per-subcycle) aggregate across **ALL** sub-cycles; **`fetchSpentOpex` (Vendors + Purpose) queries a SINGLE sub-cycle** ŌĆö the most-recent FILED one (next bullet), not all (committee adds `fetchSpentContributions`). These endpoints have no election-aggregated query mode; per-subcycle pagination is required.
- **Future-cycle handling (2026-06-11):** for an incumbent's NEXT election (e.g. a 2028 Senate cycle viewed in 2026), `/totals/?election_full` already aggregates the early fundraising but **`/candidate/{id}/committees/?cycle={electionYear}` is EMPTY** (no filing in the last 2-year sub-period yet) ŌåÆ `principal` was undefined ŌåÆ the reports fetch + all three Raised/Spent kickoffs (each gated on `if (principal)`) were skipped, hanging map/purpose/contributors/vendors in skeleton + an empty timeline. **Loading fix:** when the cycle-scoped committees query is empty, re-query unscoped and keep committees whose `cycles` overlap `subCycles` (scopes out an office-switcher's older-office PCC; one extra call only on the empty case; re-checks the fetch-race token after the await). **Opex sub-cycle scope:** `fetchSpentOpex` scopes to the most-recent FILED sub-cycle ŌĆö derived from `coverage_end_date` (the 2-year period it falls in, clamped to `subCycles`) and stored as `currentOpexSubcycle`, NOT the nominal `subCycles[last]`. Equals the last sub-cycle for complete/current cycles (no change); for a future election it's the last sub-cycle that actually has data (e.g. 2025ŌĆō2026, not the empty 2027ŌĆō2028). The `#vendors-head` title and the Spending-by-Purpose tooltip's conditional clause both label this filed sub-cycle (via `spentOpexData.opexPeriod`). Map + Top Contributors aggregate across all sub-cycles. **Future sub-cycles are dropped before querying** (`validSubCycles = subCycles.filter(y => y <= opexSubcycle)`, used for the reports fetch + `currentSubCycles` ŌåÆ the fast + slow tiers): `/schedules/schedule_a/?two_year_transaction_period={future}` returns a **404** ("data exists between 1976 and {latest}"), which would reject the slow-tier `Promise.all` and error Top Contributors ("Could not load top contributors") even though the filed sub-cycles have data. Capping at `opexSubcycle` skips the 404-ing period; the Top-Contributors card label then reads the filed range (e.g. "2023ŌĆō2026", not "2023ŌĆō2028"). The reports `.catch` already tolerated the per-sub-cycle 404 (chart was fine), and the by_state map uses no period param. Committee.html is immune ŌĆö it IS the committee (no principal lookup).
- **YTD stitching:** carries cumulative base forward across each calendar year reset within each sub-cycle, then chains sub-cycles together (chart data only).

## Domain context

- FEC "cycle" ends Dec 31 of the election year, not on election day
- House candidates file Form 3, quarterly + pre/post election reports
- Senate = 6-year terms; presidential = 4-year. X-axis logic must account for this
- `_ytd` fields reset each January 1, so a two-year cycle requires stitching year 1 final YTD + year 2 running YTD
- Memoed transactions (`memo_code: 'X'`) must be excluded from any manual totals ŌĆö they are itemization detail of other rows, not standalone money. We avoid double-counting by using FEC-computed `_ytd` fields where possible; when summing Schedule A/B rows directly (e.g. contributor aggregations), explicitly filter `d.memo_code === 'X'` before adding to the total. Conduit platforms (ActBlue, WinRed, Anedot) forward individual contributions and are reported as `memo_code='X'` rows with `entity_type='PAC'` on the recipient committee's Schedule A ŌĆö the individual donor is the main (non-memo) row on `is_individual=true` queries, and the conduit platform appears only in memos. The Top Conduit Sources table on candidate.html and committee.html (Raised tab) surfaces these memo aggregates as a distinct, legally-honest category ŌĆö "here is the money routed via X platform, representing individual donors" ŌĆö separate from Top Committee Contributors which shows committees giving their own money. Both tables draw from the same `/schedules/schedule_a/?is_individual=false` fetch; the aggregation loop splits on `memo_code === 'X'` to populate both accumulators in one pass.
- The FEC API silently ignores unrecognized query parameters ŌĆö always verify a filter is working by checking total result counts, not just response shape
- The FEC `/reporting-dates/` endpoint ignores date range params; use `report_year` + `report_type` for targeted queries (one call per year/type combination returns exactly 1 record). Note: no longer called by candidate.html ŌĆö removed 2026-04-24 as part of rate-limit reduction.
