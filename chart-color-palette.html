The three color-source layers

Colors live in three disconnected places. The timeline, the choropleth (sourced from --amber via CHART_COLORS.choroplethRgb), the monochrome purpose bars, and non-chart UI chrome flow from CSS tokens. The donut categories are raw literals, now consolidated into CATEGORY_COLORS in utils.js (Phase 1 of the color-system cleanup) so both pages share one source — but they're still literal values, not tokens, so retuning a token moves the timeline and choropleth but not the matching donut wedge (Phase 3 will promote them to --cat-* tokens). The badges throughout this page mark which layer each swatch belongs to.

Choropleth map · both pages

Per-state fill = a single token-sourced hue — --amber (#8A5F10), read at runtime via CHART_COLORS.choroplethRgb — at a continuous alpha (no discrete steps): alpha = 0.1 + (stateTotal / maxStateTotal) × 0.82, ranging 0.10 → 0.92. The top state always renders near 0.92. Both pages render the same amber ramp, independent of party. This is the one chart surface whose hue is token-driven rather than a literal.

Hover-state audit — analysis of the current approach

Read off the production code and verified empirically (reading Chart.js's resolved default-vs-active styles). The headline: the canvas charts recolor on hover; the choropleth doesn't. Timeline points and donut wedges recolor via Chart.js's built-in getHoverColor (saturate + darken) plus geometry (grow / offset); the choropleth uses a --text outline with no fill change (its shade is the data). Purpose bars and the legend are inert.

  • Hover recolors the data marks — automatically. Even though no hoverBackgroundColor is set anywhere, Chart.js auto-derives one via getHoverColor (saturate + darken): timeline points and solid donut wedges shift to a Chart.js-derived darker/saturated tone, and the donut's 1px ring brightens too. This is the effect you see on hover — it is not "geometry only."
  • Canvas recolor is Chart.js's, not ours. The canvas charts recolor via Chart.js getHoverColor (a saturate + darken transform) — automatic, not governed by a token. The choropleth deliberately does not recolor: its fill shade encodes the data, so it outlines (--text) instead of touching the fill.
  • Geometry is layered on the canvas charts only. Points grow (5 → 7), wedges offset (4px). The choropleth has no geometry change; the recolor is its entire hover.
  • Two genuine exceptions. The unitemized pattern wedge is color-exempt — getHoverColor leaves CanvasPatterns untouched, so it only offsets. And the purpose bars + donut legend have no hover at all (the legend never cross-highlights its wedge).
  • The choropleth doesn't recolor on hover. Its fill shade encodes the data, so hover is a --text outline clone (fill:none, drawn on top) — no fill change, consistent on every edge. No-data states are --surface-2; the state grid lines are --bg.

Findings (condensed)

  • Three color layers. CSS tokens (partisan, status, brand) drive the timeline, purpose bars, and UI chrome. CHART_COLORS (utils.js) reads those general tokens into JS for the timeline (the bespoke --chart-* set was retired 2026-06-03), and the choropleth reads --amber via CHART_COLORS.choroplethRgb. The donut categories are raw literals, consolidated into CATEGORY_COLORS (utils.js, Phase 1) so both pages share one source — but still literal values, not tokens, so a token retune won't propagate to the donut wedges until Phase 3 promotes them to --cat-* tokens.
  • Category color is fixed; geographic color is a single token ramp. Donut colors are assigned per-category, fixed by identity — "Individuals (itemized)" is #05234F on every page, cycle, and candidate. The choropleth is a single --amber ramp on both pages, party-independent.
  • One pattern, capacity 1. A single diagonal-stripe canvas pattern exists, used only for Individuals (unitemized) on the raised donut. There is no generator for a second distinct pattern. The legend can't render it (CSS background ≠ CanvasPattern) so it swaps to a flat legendColor — the wedge/legend asymmetry above.
  • No overflow risk. Categories come from a fixed FEC field schema, not a dynamic source, so "more categories than slots" can't happen; long-tail amounts fold into catch-alls (Other receipts, Refunds & offsets, Other Disbursements). The choropleth uses a continuous alpha scale (effectively unlimited shades of one hue).
  • Cross-page: raised donut identical; spent donut diverges (candidate 5 vs committee 8 categories); choropleth identical on both (a single --amber ramp, party-agnostic); timeline is candidate-only; purpose bars identical (monochrome).
  • Palette mid-iteration (first pass). The donut categories were recolored to a single sequential navy → parchment ramp, applied positionally so raised-slot-N and spent-slot-N share a value — retiring the old per-hue collisions (the three alpha-only greys, the multi-meaning green, the gold-vs-token "amber"). The ramp introduces its own exact-duplicate pairs (e.g. Federal funds and Contribution Refunds are both #f5e2e0); not yet finalized.
