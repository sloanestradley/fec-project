The three color-source layers

Colors live in three disconnected places. Only the timeline (and the monochrome purpose bars + non-chart UI chrome) flow from CSS tokens. The donuts and choropleth are hardcoded literals that numerically duplicate token values without referencing them — so retuning a --chart-* token moves the timeline but not the matching donut wedge. The badges throughout this page mark which layer each swatch belongs to.

Choropleth map · both pages

Per-state fill = a single party-derived hue at a continuous alpha (no discrete steps): alpha = 0.1 + (stateTotal / maxStateTotal) × 0.82, ranging 0.10 → 0.92. The top state always renders near 0.92. Hue is an inline RGB string: REP → red, DEM → blue, anything else (incl. null) → purple. Divergence: candidate party is reliably DEM/REP; committee party (c.party) is often null, so committee maps frequently fall to purple even for a clearly partisan committee.

Hover-state audit — analysis of the current approach

Read off the production code and verified empirically (reading Chart.js's resolved default-vs-active styles). The headline: the three data-mark surfaces all recolor on hover, but via two unrelated mechanisms — Chart.js's built-in getHoverColor (canvas: timeline + donuts) and a CSS brightness filter (SVG: choropleth) — with geometry (grow / offset) layered onto the canvas ones. Only the purpose bars and the legend are inert.

  • Hover recolors the data marks — automatically. Even though no hoverBackgroundColor is set anywhere, Chart.js auto-derives one via getHoverColor (saturate + darken): timeline points and solid donut wedges shift (e.g. blue #4A90D9 → #097FFD, amber #E8A020 → #EE9600), and the donut's 1px ring brightens too. This is the effect you see on hover — it is not "geometry only."
  • Two different color mechanisms that don't match. The canvas charts recolor via Chart.js getHoverColor (a saturate + darken transform); the choropleth recolors via a CSS brightness(1.35) multiplier. Both "brighten," but by different math — so hover intensity is inconsistent across surfaces, and neither is governed by a token.
  • Geometry is layered on the canvas charts only. Points grow (5 → 7), wedges offset (4px). The choropleth has no geometry change; the recolor is its entire hover.
  • Two genuine exceptions. The unitemized pattern wedge is color-exempt — getHoverColor leaves CanvasPatterns untouched, so it only offsets. And the purpose bars + donut legend have no hover at all (the legend never cross-highlights its wedge).
  • Relevance to the paused choropleth → --amber change. The brightness(1.35) hover should be re-evaluated against amber: brightening a dark, warm amber reads differently than vivid blue/red, and the faint low-density amber + brightened state may be hard to separate from the (also-brightening) no-data grey. Worth folding into that ticket's QA.

Findings (condensed)

  • Three disconnected color layers. CSS tokens (--chart-*, partisan, status) → drive the timeline, purpose bars, and UI chrome. CHART_COLORS (utils.js) mirrors the --chart-* tokens into JS for the timeline. Inline rgba/RGB literals drive the donuts and choropleth — duplicating token numbers without referencing them. A token retune will not propagate to the donuts or map.
  • Category color is fixed; geographic color is partisan. Donut and purpose-bar colors are assigned per-category, hardcoded by identity — "Individuals (itemized)" is rgba(74,144,217,1) on every page, cycle, and candidate regardless of party. Only the choropleth varies by party.
  • One pattern, capacity 1. A single diagonal-stripe canvas pattern exists, used only for Individuals (unitemized) on the raised donut. There is no generator for a second distinct pattern. The legend can't render it (CSS background ≠ CanvasPattern) so it swaps to a flat legendColor — the wedge/legend asymmetry above.
  • No overflow risk. Categories come from a fixed FEC field schema, not a dynamic source, so "more categories than slots" can't happen; long-tail amounts fold into catch-alls (Other receipts, Refunds & offsets, Other Disbursements). The choropleth uses a continuous alpha scale (effectively unlimited shades of one hue).
  • Cross-page: raised donut identical; spent donut diverges (candidate 5 vs committee 8 categories); choropleth same algorithm but committee party is often null → purple; timeline is candidate-only; purpose bars identical (monochrome).
  • Collisions: the choropleth's blue/red are the chart hues, not the partisan tag tokens; three greys distinguished only by alpha; one green and one amber each carry multiple unrelated meanings across surfaces; and several committee-spent literals (--accent, --ind) silently equal existing tokens.
